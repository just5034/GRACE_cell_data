# GRACE Paper 2 — Data Availability

This repository hosts the per-benchmark artifacts for the three-arm
architecture ablation reported in the GRACE Paper 2 manuscript
(_Cellular GRACE: ..._). All three arms were run on the same thirteen-benchmark
Celeritas-derived suite at 10,000 events per simulation per energy point,
with the same model (Claude Sonnet 4) and the same prompts; they differ only
in which agent architecture and which infrastructure components were active.

## Layout

```
cellular_qe_off/   Cellular arm with the quality-engineering stack OFF
                   (no reference-detector registry, no expected-range injection,
                    no hard-citing, no fresh-context evaluation, no per-step redo)

cellular_qe_on/    Cellular arm with the full quality-engineering stack ON

monolithic/        Monolithic GRACE arm (single ~900-line prompt per step,
                   no cellular decomposition, no QE stack)
```

Inside each arm folder, the per-benchmark sub-folders follow the convention
`<benchmark_name>_<model>_<timestamp>/` and contain:

| File | What it is |
|---|---|
| `benchmark_result.json` | Top-level result summary: success flag, error string, task confidence, plan, duration |
| `academic_report.md` | Agent-generated final report for the run |
| `scientific_memory.json` | Agent's accumulated reasoning/observations across the run (sometimes gzipped, see below) |
| `reasoning_trace.json` | Step-by-step trace of every cell invocation, tool call, and recovery decision |
| `*.gdml`, `*.mac` | Geometry files and Geant4 macros generated by the agent |
| `*.png`, `*.pdf` | Plots produced by the agent's analysis code |
| `*.py` | LLM-generated analysis scripts that the agent wrote and executed |
| `*_events.parquet` | Per-event Geant4 summary (eventID, totalEdep, nHits) |
| `token_usage.jsonl` (cellular only) | Per-call Anthropic token counts |
| `.status_<bench>` | SLURM-level PASS/FAIL sentinel |

## Reduced artifacts

Raw per-hit parquet files (`*_hits_data.parquet`) and ROOT files
(`*_hits.root`) were dropped during archive; they regularly run to multi-GB
per benchmark. They are regeneratable from the corresponding GDML, Geant4
macro, and RNG seed via the same git commit of the GRACE codebase. The
small artifacts retained here (reports, JSON, plots, GDMLs, macros,
generated Python, per-event parquets, token logs, sentinels) are sufficient
to reproduce every claim in the paper.

## Gzipped files

One file in `cellular_qe_off/` exceeds GitHub's 100 MB hard per-file limit
when uncompressed and has been gzipped:

```
cellular_qe_off/tracking/tracking_claude-sonnet-4-20250514_20260427_043455/scientific_memory.json.gz
```

To decompress:
```bash
gunzip cellular_qe_off/tracking/.../scientific_memory.json.gz
```

The gzip is round-trip lossless; the resulting JSON is byte-identical to
the original artifact written by the agent.

The cellular QE-off and QE-on jobs are paired arms of a single ablation
designed for direct comparison; the monolithic arm was run later under the
same prompts and event count for the apples-to-apples three-arm comparison.

## Pass-rate summary (from the paper)

| Arm | PASS (raw) | PASS (honest, infra failures excluded) |
|---|---|---|
| Monolithic | 2/13 (15%) | 2/13 (15%) |
| Cellular QE-off | 4/13 (31%) | 4/10 (40%) |
| Cellular QE-on | 5/13 (38%) | 5/12 (42%) |

"Honest" pass rate excludes infrastructure failures (OOM during ROOT-to-Parquet
conversion of multi-GB hit files, Anthropic API rate-limits) where the agent
never produced a `benchmark_result.json`. The monolithic arm has no
infrastructure failures because it post-dates the `glibc` arena-release fix;
its failures are all agent-decision failures.

## Notes on what's missing

- **Monolithic muon run**: the per-bench artifacts for `monolithic/muon_*/`
  are empty in this repository because the agent did not produce them,
  despite emitting `success=True` in the SLURM-level summary. This is itself
  a sharp instance of the report-stage decoupling discussed in section 4.4
  of the paper: the run-level success sentinel was emitted in the absence
  of the simulation outputs that would substantiate it. The run-level
  signals (`success=True`, 7/7 steps, 434 s, `task_confidence`=0.95) are
  recovered from the SLURM-level `benchmark_summary.json` and reported in
  section 4.3.

- **Five OOM-killed monolithic runs** (sampling_ecal, geometry_comparison,
  shower_containment, calorimeter, timing_calorimeter from job 17996548)
  were re-run in jobs 18010045 and 18023199 after the `malloc_trim` and
  SLURM-mem fixes. The per-bench folders in this repository hold the
  successful re-runs, not the OOM'd originals.
